    CSP(Content-Securiy-Policy)가 생겨난 이유

    SOP/CORS는 자바스크립트가 리소스에 접근하지 못하게 하기위해 만들었다면 CSP(Content-Securiy-Policy)는 HTML태그를 삽입해 스크립트와 같은 실행을 막기 위해 만들어졌습니다.

    만약 HTML상에 데이터를 삽입하는 종류의 공격이 발생하였을 때 피해를 줄이고 웹 관리자가 공격 시도를 보고받을 수 있도록 하기 위함입니다.

    CSP(Content-Securiy-Policy) 지시어

    CSP의 지시어는 웹 애플리케이션에서 다양한 리소스(스크립트, 스타일, 이미지 등)의 로드를 제어하기 위해 사용됩니다.

    지시어는 각각의 리소스에 대해 허용할지 설정하고, 세미콜론(;)을 이용하여 지시어를 나눌 수 있습니다.

    주요 지시어(src)

    1. default-src : CSP에 사용되는 모든 유형의 리소스에 대한 기본으로 출처를 설정합니다.
    2. script-src : 자바스크립트의 출처를 설정합니다.
    3. style-src : CSS의 출처를 설정합니다.
    4. img-src : 이미지의 출처를 설정합니다.
    5. font-src : 폰트의 출처를 설정합니다.

    connect-src : XMLHttpRequest, Fetch API, WebSocket, EventSource와 같은 네트워크 요청의 출처를 설정합니다.
    object-src : <object>, <embed>, <applet> 태그에서 사용하는 리소스의 출처를 설정합니다.
    media-src : 오디오 및 비디오 파일의 출처를 설정합니다.
    frame-src : <iframe> 및 <frame>에서 로드하는 리소스의 출처를 설정합니다.
    worker-src : Web Workers와 Shared Workers에서 사용하는 리소스의 출처를 설정합니다.
    manifest-src : 웹 애플리케이션 매니페스트 파일의 출처를 설정합니다.

    그 외 지시어

    report-uri : CSP 위반이 발생했을 때, 이를 서버로 보고할 URL을 정의합니다.
    base-uri : HTML의 <base> 태그에서 사용하는 URI의 출처를 정의합니다.
    form-action : <form> 태그에서 action 으로 제출 될 때, 어느 도메인으로 데이터를 전송할 수 있는지 정의합니다.
    frame-ancestors : 현재 페이지가 어떤 도메인에서 <frame>이나 <iframe>으로 삽입될지 제어합니다.(클릭재킹 방지)
    upgrade-insecure-requests : 페이지에 있는 HTTP 리소스를 자동으로 HTTPS로 업그레이드합니다.
    block-all-mixed-content : HTTPS 페이지에서 HTTP 리소스(혼합 콘텐츠)의 로드를 완전히 차단합니다.

    CSP(Content-Securiy-Policy) 주요 키워드 종류

    'self' : 현재 출처에서만 리소스를 허용합니다.
    'none' : 해당 리소스 유형을 차단합니다.
    'unsafe-inline' : 인라인 스크립트나 스타일 허용합니다.
    'unsafe-eval' : eval() 등 위험한 자바스크립트 실행 허용합니다.
    'strict-dynamic' : 동적 스크립트 로드에 대한 신뢰 상속합니다.
    'nonce-{value}' : 특정 nonce 값을 가진 스크립트만 허용합니다.
    'sha256-{hash}' : 특정 해시 값을 가진 스크립트만 허용합니다.
    도메인 지정 : 특정 도메인에서만 리소스를 로드 허용합니다.
    https: : HTTPS를 사용하는 모든 출처에서 리소스를 허용합니다.
    http: : HTTP를 사용하는 모든 출처에서 리소스를 허용합니다.
    data: : 스키마를 사용하여 인라인으로 리소스를 직접 포함할 수 있습니다.
    blob: : Blob URL로 생성된 리소스를 허용합니다.
    * : 모든 출처에서 리소스를 허용합니다.

    XSS를 막기위한 CSP 기본 사용 방법

    XSS를 방지하기위해 CSP로 script-src를 사용하여 차단하는 방법을 알려드리겠습니다.

    두가지 방법이 있습니다.

    1. 서버언어로 설정하는 방법(예시는 PHP로 들겠습니다)
    <?php $nonce = hash('sha256', microtime()); 
header("Content-Security-Policy: script-src 'self' 'unsafe-inline' 'nonce-{$nonce}' 'strict-dynamic';"); ?>
<script nonce="<?=$nonce?>">alert(1);</script>
    1. 메타태그로 설정하는 방법
    <?php $nonce = hash('sha256', microtime()); ?>
<meta http-equiv="Content-Security-Policy" content="script-src 'self' 'unsafe-inline' 'nonce-<?=$nonce?>' 'strict-dynamic';">
<script nonce="<?=$nonce?>">alert(1);</script>

    두가지 방법 차이점

    서버언어로 CSP를 설정할 시 HTTP 헤더에 포함되어 있습니다.

    메타태그로 CSP를 설정할 시 HTTP 헤더에 없습니다.

    CSP 설정 설명

    일단 지시어는 script-src하나만 사용 하였으며 키워드로는 'self', 'unsafe-inline', 'nonce-hash', 'strict-dynamic'를 사용하였습니다.

    script-src는 자바스크립트의 출처를 설정하기 위해 지시어로 사용하였고 키워드도 하나씩 살펴보겠습니다.

    'self'를 사용하여 현재 출처에서만 script를 허용합니다.
    'unsafe-inline'를 사용하여 인라인으로 된 스크립트의 사용을 허용합니다.
    'nonce-hash'를 사용한 이유는 'unsafe-inline'와 'strict-dynamic'를 사용하면 CSP가 스크립트의 출처를 엄격하게 보기 때문에 nonce설정을 한 특정 스크립트에만 실행되게 허용합니다.
    'strict-dynamic'를 사용하여 동적으로 생성되는 스크립트에 대해 신뢰하여 개발 시 좀 더 유연한 개발을 할수 있게 허용합니다.

    이렇게 CSP설정 하면 <script>사용 시 <script nonce="<?=$nonce?>">alert(1);</script>이렇게 nonce설정을 해줘야 합니다.

    블로그를 봐주셔서 감사합니다.
