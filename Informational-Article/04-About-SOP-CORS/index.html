    SOP/CORS가 생겨난 이유

    해커가 XSS 취약점이나 피싱사이트를 사용하여 다른 사이트에서 악성코드를 심은 뒤 사용자가 이를 클릭하면 서버로 강제적인 요청이 이루어지기 때문에 해당 요청이 사용자의 요청인지 악성코드의 요청인지 구분할 수 없기 때문에 이 점을 보안하기위해 만들어 진 것이 SOP/CORS 라는 웹 보안 정책입니다

    SOP(Same-Origin-Policy)

    브라우저가 웹 페이지에서 다른 출처(Origin)의 자원에 접근하지 못하도록 제한하는 강제적인 규칙이며

    출처(Origin)는 Potocol + Domain + Port 3가지가 모두 같으면 Same-Origin로 자바스크립트에서 리소스에 접근 할 수 있습니다

    만약 하나라도 다르면 Cross-Origin로 자바스크립트에서 다른 출처의 리소스에 접근하지 못하도록 제한합니다.

    해당 규칙 덕분에 해커가 심어놓은 악성코드가 실행하여 개인 정보를 가로채는 것을 막을 수 있게 되었지만

    SOP정책으로 인해 웹 사이트의 확장성이 매우 떨어지게 되어 Cross-Origin일때도 다른 출처의 리소스에 접근 할 수 있도록 CORS라는 정책을 하나 더 만들게 됩니다.

    CORS(Cross-Origin-Resource-Sharing)

    몇 가지 규칙에 해당되면 다른 출처(Origin)에서도 자원에 대해 접근 할 수 있도록 허락하는 규칙입니다

    SOP가 무조건 허용 안되게 막는다면 CORS는 SOP에게 허락 할 목록들은 허용을 하게 해달라는 목록들을 적은 규칙입니다.

    서버에서 헤더에 ACAO(Access-Control-Allow-Origin)등 허용할 목록을 만들어서 설정합니다.

    CORS(Cross-Origin-Resource-Sharing) 동작 시나리오

    1. 사용자의 브라우저는 요청 해더에 Origin이라는 필드에 어디에서 요청을 하는 지 담아서 전송합니다(만약 Same-Origin일 경우 헤더에 Origin필드는 보이지 않고 Cross-Origin일 때 보입니다.)

    2. 서버는 이 요청에 대한 응답 헤더에 ACAO(Access-Control-Allow-Origin)이라는 필드에 허용된 출처를 담아서 다시 사용자의 브라우저로 전송합니다.

    3. 브라우저는 자신이 보냈던 요청 헤더의 Origin과 ACAO(Access-Control-Allow-Origin)를 비교 하여 자바스크립트에서 리소스를 접근을 허용 할 지 결정합니다.

    CORS 설정 사진

    CORS(Cross-Origin-Resource-Sharing) 설정

    결국 SOP에 대한 허용을 위해 CORS를 사용하려면 설정이 필요하기 때문에 설정을 잘 하는 것이 중요합니다.

    설정은 허용할 리스트들을 적은 일종의 White List Filter 작업이라 볼 수 있습니다.

    1. ACAO(Access-Control-Allow-Origin) : 와일드 카드(*)를 사용하여 모든 사이트를 허용 할 수 있지만 ACAC가 true일 경우 와일드 카드(*)사용이 불가능 합니다. (* 또는 https://naver.com처럼 사이트 주소)

    2. ACRM(Access-Control-Request-Methods) : 리소스 접근을 허용하는 HTTP메서드 지정합니다. (GET, POST, PUT, DELETE)

    3. ACAH(Access-Control-Allow-Headers) : 요청을 허용할 헤더를 지정합니다. (Origin,Accept,X-Requested-With,Content-Type,Access-Control-Request-Method,Access-Control-Request-Headers,Authorization)

    4. ACAC(Access-Control-Allow-Credentials) : 클라이언트 요청에서 자격증명(쿠키,HTTP 인증 정보, TLS 클라이언트 인증서)을 포함 해야하는 경우 true를 사용합니다.

    5. ACMA(Access-Control-Max-Age) : 서버에 요청하기 전에 요청이 유효한지 확인하기 위해 OPTIONS 메서드로 예비 요청을 보내 Preflight 요청을 캐싱 시켜 최적화합니다. (1당 1초)

    6. ACEH(Access-Control-Expose-Headers) : 기본적으로 브라우저에게 노출이 되지 않지만, 브라우저 측에서 접근할 수 있게 허용해주는 헤더를 지정 합니다. (Content-Length)

    CORS 모두 설정된 사진

    CORS 보안 비하인드

    개발자들이 CORS설정을 매번 하는 것이 귀찮아서 처음에 다 열어 두기 위해 ACAO에 와일드 카드(*)를 쓰기 시작하자 브라우저 보안은 이건 아니라 생각 하였습니다.

    그래서 브라우저 보안은 타 도메인에서 쿠키를 포함해서 요청하는 것에 대한 응답은 ACAC를 만들어서 true일 때만 가능하게 만든 후 ACAC가 true일 때는 ACAO에 와일드 카드(*)를 사용하지 못하게 하였습니다.

    하지만 몇몇 개발자들은 그냥 요청 받은 Origin을 ACAO에 넣어버린 뒤 브라우저에 전송해버리는 것으로 와일드 카드(*)대신 사용해 버렸습니다.

    해당 방식으로 만든 웹 사이트는 취약한 CORS에 속하며 이처럼 CORS 설정이 올바르게 설정되어 있지 않다면 모의해킹 시 보고하는것도 중요합니다.
