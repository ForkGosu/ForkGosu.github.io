    JWT(Json Web Token)란?

    JWT(Json Web Token)란 Json 포맷을 이용하여 인증 및 정보교환을 목적으로 사용되는 토큰입니다.

    사용자의 신원을 확인하거나, 서버 간에 정보를 전송할 때 사용됩니다.

    JWT(Json Web Token)의 구성요소

    JWT는 세가지 부분(Header.Payload.Signature)으로 나뉘며 각각 점(.)을 기준으로 구분합니다.

    헤더(Header)
    JWT가 어떤 알고리즘을 사용하여 서명했는지에 대한 정보가 담겨있습니다.

    페이로드(Payload)
    JWT에 담고 싶은 데이터 모음입니다. 사용자ID, 권한, 만료시간 등이 담겨있습니다.

    서명(Signature)
    JWT의 무결성을 보장하기 위해 사용됩니다. 헤더와 페이로드를 서버의 비밀키로 서명하여 클라이언트가 해당 JWT를 조작하였는지 검사할 수 있습니다.

    JWT(Json Web Token) 원리

    만약 다음과 같은 JWT 데이터가 있다면 다음과 같이 Decoded가 됩니다.

    eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

    맨 마지막 서명(Signature) 부분은 비밀키로 암호화 한 것이기 때문에 base64로 디코딩 할 수 없습니다.

    그 이유는 헤더와 페이로드가 수정되었는지 아닌지에 대한 무결성을 확인하기 위한 것이기 때문입니다.

    만약 다른 JWT를 확인해보고 싶다면 jwt.io에 들어간 후 JWT를 입력하면 JWT에 담긴 정보들을 볼 수 있습니다.

    JWT(Json Web Token) 장점

    JWT는 클라이언트 측에서 저장된 데이터를 가지고 확인하기 때문에 서버에 데이터를 저장하지 않아도 되고, 그래서 Session과는 다르게 DB에 접근하지 않아 서버측의 부하를 낮출 수 있다는 장점이 있습니다.

    만약 분산 마이크로서비스를 많이 운용중이라면 중앙 집중식의 인증 서버와 DB에 의존하지 않고 각각의 서비스에서 쉽게 인증할 수 있다는 장점이 있습니다.

    JWT(Json Web Token) 주의사항

    alg방식을 none으로 할 경우 아예 보안이 되지 않기 때문에 알고리즘 방식은 none을 하지 말아야 합니다.

    Signature의 시크릿 키를 너무 쉽게 작성하면 Brute Force를 사용하여 JWT를 만들 수도 있기 때문에 시크릿 키는 무조건 길고 알 수 없도록 설정해야 합니다.

    JWT가 도난 당할 수도 있기 때문에 HttpOnly설정을 하고, 유효기간을 짧게 한 뒤 Refresh token을 사용하고, Refresh token rotation을 사용해서 Refresh token은 무조건 1회용으로만 사용 할 수 있도록 합니다.

    토큰 자체에 정보를 담고 있기 때문에 패스워드 정보나 중요한 정보는 저장하지 않는게 좋습니다.

    JWT는 상태를 저장하지 않기 때문에 한번 만들어지면 삭제하거나 수정하는 것이 불가능 하기 때문에 반드시 토큰 만료 시간을 넣어주어야 합니다.

    JWT(Json Web Token) 결론

    JWT는 보안을 위해 주의해야 할 사항들이 많이 고려해야 합니다.

    그래서 DB에 회원이 1억명이거나 분산 마이크로서비스들이 많다면 JWT를 사용하고 그렇지 않다면 Session방식을 사용하는 것이 좋을 수도 있을 것 같습니다.
